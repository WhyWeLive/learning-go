# Git workflow (learning-go)

Заголовки Issues/PR — **русский**; slug ветки — **английский** kebab-case.

## Единица поставки

**По умолчанию: 1 epic ≈ 1 ветка ≈ 1 PR.**

- Child-tasks (#2, #3, …) — учебные срезы внутри epic (теория + практика + AC), не обязанность плодить PR на каждую.
- В теле PR: `Closes #<epic>` и перечисление закрываемых tasks (`Closes #2, #3, …` или checklist).
- Отдельный PR на task — только если ученик сам дробит или diff реально раздулся; согласовать в чате.

Цель процесса — **научиться и получить строгий review по сути**, не набрать ритуал merge’ей.

## Ветки

| Префикс | Когда |
|---|---|
| `feat/#<n>-<slug>` | продуктовая фича |
| `fix/#<n>-<slug>` | баг |
| `chore/#<n>-<slug>` | инфраструктура / tooling / учебный каркас |
| `docs/#<n>-<slug>` | документация |
| `refactor/#<n>-<slug>` | рефакторинг без смены поведения |

Правила:

- `<n>` — номер **epic** (предпочтительно) или task, если отдельный PR согласован.
- `<slug>` — короткий kebab-case **на английском**.
- Ветка от актуального `main`.
- Коммиты напрямую в `main` — запрещены по конвенции.
- При `status:in-progress` на epic/активной работе имя ветки фиксируется в Мета.

Пример фазы 0: epic `#1` → `chore/#1-marketplace-skeleton`.

## Коммиты

Conventional Commits: `feat:`, `fix:`, `chore:`, `test:`, `docs:`, `refactor:`.

В теле/футере: `Refs #N` на промежуточных коммитах; `Closes` — в PR.

Дробление на осмысленные коммиты внутри одной ветки **приветствуется** (module → layout → health → docker).

## Pull Request

- Заголовок: `#<epic> Краткий заголовок на русском`
- Base: `main`
- Один PR закрывает epic (и связанные tasks), пока не согласовали иное.

Тело (минимум):

```markdown
## Связь
Closes #1
Closes #2, #3, …  <!-- tasks фазы, которые реально закрыты diff’ом -->

## Что сделано
…

## Как проверить
…  <!-- команды и сценарии — пишет ученик; это доказательство, не копипаста из Issue -->

## Учебный итог
Ответы на контрольные вопросы из Theory-секций закрываемых Issues.
Строгие формулировки: определения, границы понятий, чем X не является.

## Риски / известные ограничения
…
```

## Статусы Issue ↔ git

- Tasks могут оставаться `status:ready` / отмечаться в epic-checklist по мере работы в одной ветке.
- Epic → `status:in-progress` при старте работы; при открытии PR → `status:review`; после merge → close epic и закрытых tasks.
- Агент не мержит и не пушит application code; review — `gh pr review` (`reviewer.md`).

## Review: фокус

Проверять суть AC и теорию в «Учебный итог». Не устраивать бюрократию ради отдельных PR на каждый task, если поставка идёт epic-PR.
